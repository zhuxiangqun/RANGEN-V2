# 推理链生成提示词对比分析

生成时间: 2025-12-04

## 一、DeepSeek建议的模板特点

### 1.1 核心优势

1. **模块化设计**
   - 分阶段：问题分析 → 逐步求解 → 整合验证 → 最终输出
   - 每个阶段有明确的检查点和验证步骤
   - 结构化输出格式

2. **自适应策略**
   - 根据问题类型自动选择推理方法
   - 针对不同领域有专用子模板
   - 动态调整推理策略

3. **严谨性要求**
   - 强制外化思考过程
   - 包含逻辑一致性检查
   - 证据可靠性评估
   - 不确定性说明

4. **验证机制**
   - 每个子问题有检查点
   - 最终整合验证
   - 敏感性分析

### 1.2 关键要素

- **问题分析与拆解**：明确问题类型、关键要素、子问题链
- **逐步求解**：每个子问题有策略、执行、检查点、结论
- **整合验证**：逻辑一致性、证据可靠性、敏感性分析
- **最终输出**：推理链总结、最终答案、不确定性说明

## 二、当前系统的提示词特点

### 2.1 当前优势

1. **强调可执行性**
   - 要求生成可检索的子查询
   - 强调子查询格式（单一、纯问题）
   - 禁止包含答案和推理过程

2. **提供明确示例**
   - 展示错误做法（❌ WRONG）
   - 展示正确做法（✅ CORRECT）
   - 针对具体问题类型给出示例

3. **拆解要求**
   - 禁止简单重复原问题
   - 禁止合并多个问题
   - 要求为每个条件创建单独步骤

### 2.2 当前不足

1. **缺乏阶段划分**
   - 没有明确的问题分析阶段
   - 没有验证和整合阶段
   - 缺少检查点和验证机制

2. **缺乏自适应策略**
   - 没有根据问题类型选择推理方法
   - 没有针对不同领域的专用策略
   - 推理策略相对固定

3. **缺乏严谨性要求**
   - 没有逻辑一致性检查
   - 没有证据可靠性评估
   - 没有不确定性说明

4. **输出格式不够结构化**
   - 只要求JSON格式
   - 没有详细的推理过程展示
   - 缺少验证步骤

## 三、对比分析

| 方面 | DeepSeek模板 | 当前系统 | 相似度 |

| 问题拆解 | ✅ 要求拆解为子问题 | ✅ 要求拆解为子问题 | 高 |

| 子查询格式 | ⚠️ 未明确强调 | ✅ 明确要求纯问题格式 | 中 |
| 示例提供 | ✅ 提供示例 | ✅ 提供正确/错误示例 | 高 |

### 3.2 差异之处

| 方面 | DeepSeek模板 | 当前系统 | 差异 |
|------|-------------|---------|------|
| 阶段划分 | ✅ 4个明确阶段 | ❌ 无明确阶段 | 大 |
| 问题类型识别 | ✅ 自动识别和适配 | ❌ 无类型识别 | 大 |
| 验证机制 | ✅ 多层级验证 | ❌ 无验证机制 | 大 |
| 不确定性说明 | ✅ 要求说明置信度 | ❌ 无要求 | 中 |
| 输出格式 | ✅ 结构化Markdown | ⚠️ 简单JSON | 中 |
| 检查点 | ✅ 每个子问题有检查点 | ❌ 无检查点 | 大 |

## 四、改进建议

### 4.1 融合DeepSeek模板的优势

1. **添加阶段划分**
   - 问题分析与拆解阶段
   - 逐步求解阶段
   - 整合验证阶段
   - 最终输出阶段

2. **添加问题类型识别**
   - 自动识别问题类型
   - 根据类型选择推理策略
   - 针对不同类型提供专用指导

3. **添加验证机制**
   - 每个子问题的检查点
   - 逻辑一致性检查
   - 证据可靠性评估

4. **改进输出格式**
   - 保持JSON格式（便于程序处理）
   - 添加详细的推理过程字段
   - 添加验证和置信度信息

### 4.2 保留当前系统的优势

1. **保持可执行性要求**
   - 继续强调子查询必须是可检索的纯问题
   - 禁止包含答案和推理过程
   - 保持格式要求

2. **保持示例指导**
   - 继续提供正确/错误示例
   - 针对具体问题类型给出示例

3. **保持拆解要求**
   - 继续禁止简单重复原问题
   - 继续要求为每个条件创建单独步骤

## 五、改进后的提示词结构建议

### 5.1 融合版本结构

```
1. 角色设定和核心指令
2. 第一阶段：问题分析与拆解
   - 问题理解与分类
   - 关键要素提取
   - 问题拆解（保留当前要求）
3. 第二阶段：逐步求解
   - 每个子问题的解决策略
   - 具体执行（生成可执行的子查询）
   - 检查点
4. 第三阶段：整合验证
   - 逻辑一致性检查
   - 证据可靠性评估
5. 第四阶段：最终输出
   - 推理链总结
   - 最终答案
   - 不确定性说明
6. 输出格式要求（JSON + 详细字段）
7. 示例（保留当前正确/错误示例）
```

### 5.2 关键改进点

1. **添加问题类型识别和自适应策略**
2. **添加验证机制和检查点**
3. **改进输出格式，包含更多元数据**
4. **保持可执行性要求**
