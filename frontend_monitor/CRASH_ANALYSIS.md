# 系统崩溃原因分析报告

## 问题描述
**直接运行核心系统**（`scripts/run_core_with_frames.py`）处理101个样本时，整个电脑崩溃重启。

## 最新崩溃信息（2025-01-XX）

### 崩溃类型：macOS 内核 Watchdog Timeout
```
panic(cpu 5 caller 0xfffffe001bada40c): watchdog timeout: no checkins from watchdogd in 93 seconds
```

**关键信息**：
- **崩溃原因**：系统在93秒内完全没有响应，触发 macOS 内核 watchdog 保护机制
- **系统级别**：这是内核级别的崩溃，不是 Python 程序崩溃
- **CPU 使用**：kernel_task 占用大量 CPU (4938760)
- **影响范围**：整个系统崩溃重启

**崩溃机制**：
1. 系统负载过高，导致内核无法响应 watchdog 检查
2. 93秒内没有任何进程能够响应系统调用
3. macOS 内核判定系统死锁，触发 panic 保护机制
4. 系统强制重启以保护数据完整性

## 问题定位
用户明确说明是"直接运行核心系统时出现的现象"，而非通过前端监控系统运行。因此问题出在核心系统本身的代码中。

## 根本原因分析（直接运行核心系统）

### 1. **严重的内存泄漏 - 超时处理时读取整个日志文件** ⚠️ 最严重

**问题位置**：
- `scripts/run_core_with_frames.py:314-337` - 超时异常处理中


**影响**：
- 对于101个样本，日志文件可能达到 **几十MB甚至上百MB**
- 每次超时都会读取整个日志文件到内存
- 如果多个样本超时，会多次读取整个文件
- 对于101个样本，如果有10个超时，就会读取10次整个日志文件
- **内存使用量 = 日志文件大小 × 超时样本数 × 对象开销**
- **可能导致内存使用量达到数GB，触发系统 OOM (Out of Memory)**

### 2. **内存泄漏 - 每个样本处理时累积数据** ⚠️ 严重

**问题位置**：
- `scripts/run_core_with_frames.py:141-345` - `run()` 函数

**问题描述**：

  - `reasoning_steps`: 推理步骤列表
  - `evidence`: 证据列表

- 这些数据在循环中累积，直到所有样本处理完成
- **Python 垃圾回收器可能无法及时清理**

### 3. **LLM API 调用导致的内存和资源压力** ⚠️ 严重

**问题描述**：

  - 创建HTTP连接
  - 发送请求（可能包含大量上下文）
  - 接收响应（可能包含大量文本）
  - 解析和存储响应数据

**影响**：
- **网络连接数**：可能达到系统限制
- **内存占用**：每个LLM响应可能包含几千到几万字
- **文件描述符**：HTTP连接占用文件描述符
- **系统资源耗尽**：可能导致系统崩溃

### 4. **日志文件无限增长** ⚠️ 中等

**问题位置**：
- `evaluation_system/research_logger.py` - 日志记录

**问题描述**：
- 每个样本产生大量日志（推理过程、LLM调用、错误信息等）

- 日志文件持续增长，没有轮转或清理机制

**影响**：
- 磁盘空间占用
- 读取日志文件时内存占用（如超时处理时）

### 5. **异步任务和资源未及时清理** ⚠️ 严重（导致 Watchdog Timeout）

**问题位置**：
- `src/unified_research_system.py` - 统一研究系统
- `src/core/llm_integration.py` - LLM 集成层
- `src/utils/unified_jina_service.py` - Jina 服务

**问题描述**：
- 每个样本处理时创建大量异步任务
- 虽然最后有 `await system.shutdown()`，但在处理过程中：
  - 异步任务可能累积
  - 资源（连接、缓存等）可能未及时释放
  - 只有在所有样本处理完成后才清理

**具体问题**：
1. **HTTP 连接未及时关闭**：
   - 每个 LLM API 调用创建新的 HTTP 连接
   - 连接可能没有正确关闭，导致连接池耗尽
   - 101个样本 × 每个样本10次调用 = 1010个连接
   - 如果连接没有及时关闭，可能达到系统文件描述符限制

2. **异步任务累积**：
   - `_active_queries` 字典虽然会清理，但在高并发下可能累积
   - 超时任务可能没有正确取消
   - 事件循环中可能有大量 pending 任务

3. **缓存无限增长**：
   - `_cache_system` 中的缓存字典会不断累积
   - 虽然有 `max_cache_size=1000` 限制，但可能没有及时清理过期缓存
   - 101个样本可能产生大量缓存条目

4. **事件循环阻塞**：
   - 大量同步操作可能阻塞事件循环
   - 垃圾回收可能阻塞事件循环
   - 文件 I/O 操作可能阻塞事件循环

**影响**：
- 内存占用持续增长
- 资源（文件句柄、网络连接）可能耗尽
- **系统资源完全耗尽，导致 Watchdog Timeout**

## 崩溃机制

### 机制1：Watchdog Timeout（最新发现）⚠️ 最严重

**触发条件**：
- 系统在93秒内完全没有响应
- 所有进程（包括内核任务）都无法响应系统调用
- macOS 内核 watchdog 检测到系统死锁

**根本原因**：
1. **系统资源完全耗尽**：
   - CPU 100% 使用率，所有核心都被占用
   - 内存耗尽，系统开始频繁交换（swap）
   - 磁盘 I/O 饱和，无法响应任何读写请求
   - 文件描述符耗尽，无法创建新的连接或文件

2. **死锁或无限循环**：
   - 大量异步任务相互等待资源
   - HTTP 连接池耗尽，新请求无法建立连接
   - 事件循环阻塞，无法处理新的任务
   - 垃圾回收器无法运行（因为系统资源耗尽）

3. **级联故障**：
   - 一个资源耗尽导致其他资源也无法释放
   - 系统进入死锁状态，无法恢复
   - 内核无法调度任何进程，包括 watchdogd

**101个样本的处理压力**：
- 每个样本：5-10次 LLM API 调用
- 总调用数：500-1000次 HTTP 请求
- 每个请求：创建连接、发送数据、接收响应、解析JSON
- 并发限制：虽然设置了 `max_concurrent_queries=3`，但底层可能有更多并发
- **累积效应**：101个样本 × 每个样本30分钟超时 = 可能运行数小时

### 机制2：内存耗尽

1. **内存耗尽**：
   - 日志文件不断增长（101个样本可能产生几十MB日志）
   - 频繁的全量读取导致内存使用量指数级增长
   - 系统内存耗尽，触发 OOM Killer
   - macOS 可能直接重启系统以保护数据

2. **资源耗尽**：
   - CPU 100% 使用率
   - 磁盘 I/O 饱和
   - 系统无法响应，可能触发内核保护机制

## 修复方案

### 方案1: 增量日志解析（推荐）✅

**核心思路**：只解析新增的日志内容，而不是每次都读取整个文件。

**实现**：
1. 记录上次解析的位置（文件偏移量或行号）
2. 只读取新增的内容
3. 增量更新样本和智能体调用数据

### 方案2: 减少解析频率

**实现**：
1. SSE 流改为每2-5秒解析一次，而不是0.5秒
2. 进程输出中，每10-20行才读取一次日志文件，而不是每行都读

### 方案3: 限制日志文件大小

**实现**：
1. 实现日志轮转（log rotation）
2. 当日志文件超过一定大小（如10MB）时，归档旧日志
3. 只解析最新的日志文件

### 方案4: 优化内存使用

**实现**：
1. 使用生成器而不是列表
2. 及时释放不需要的对象
3. 限制解析的样本数量（如只解析最近100个样本）

## 紧急修复建议（针对 Watchdog Timeout）

### P0 - 立即修复（防止系统崩溃）

1. **每处理完一个样本后立即清理资源**：
   - 在每个样本处理完成后，立即清理该样本相关的资源
   - 关闭 HTTP 连接
   - 清理缓存中的过期条目
   - 强制垃圾回收

2. **限制并发数**：
   - 将 `MAX_CONCURRENT_QUERIES` 降低到 1（串行处理）
   - 虽然会变慢，但可以避免资源耗尽

3. **分批处理**：
   - 将101个样本分成多个批次（如每批10个）
   - 每批处理完后，完全关闭系统并重新初始化
   - 给系统时间释放资源

4. **增加资源监控**：
   - 监控内存使用、文件描述符数量、CPU 使用率
   - 当资源使用超过阈值时，主动清理或暂停处理

### P1 - 短期修复（1-2天）

1. **实现连接池管理**：
   - 使用连接池复用 HTTP 连接
   - 设置连接池大小限制
   - 自动关闭空闲连接

2. **优化缓存清理**：
   - 实现 LRU 缓存淘汰策略
   - 定期清理过期缓存
   - 限制缓存总大小

3. **异步任务管理**：
   - 限制同时运行的异步任务数量
   - 及时取消超时任务
   - 监控 pending 任务数量

4. **减少日志文件读取频率**：
   - 实现增量日志解析
   - 减少日志文件读取频率

### P2 - 长期优化（1周）

1. **实现日志轮转**：
   - 当日志文件超过一定大小时自动轮转
   - 只解析最新的日志文件

2. **优化内存使用**：
   - 使用生成器而不是列表
   - 及时释放不需要的对象
   - 限制解析的样本数量

3. **系统资源监控和告警**：
   - 实时监控系统资源使用情况
   - 当资源使用超过阈值时发出告警
   - 自动降级或暂停处理
