# 系统架构优化与重构方案

## 现状分析：当前系统的痛点
经过对代码库的深入扫描，发现当前系统 (`src/`) 存在明显的**组件膨胀**和**职责重叠**问题，这正是你提供的优化架构图所要解决的核心痛点。

### 1. "形容词膨胀" (Adjective Inflation)
代码中存在大量命名相似、功能重叠的组件，例如：
- **Router**: `EntryRouter`, `IntelligentRouter`, `LangGraphConfigurableRouter`
- **Context**: `ContextManager`, `IntelligentContextManager`, `AdvancedContextManager`, `EnhancedContextEngineering`
- **Reasoning**: `ReasoningExpert`, `RealReasoningEngine`, `AdvancedReasoningEngine`, `AdvancedReasoningEngineV2`
- **Orchestrator**: `UnifiedResearchSystem`, `IntelligentOrchestrator`, `ExecutionCoordinator`

**结论**：目前的系统在不断的迭代中堆砌了大量"Advanced"、"Intelligent"、"Enhanced"前缀的类，导致核心逻辑分散，维护困难。

### 2. 架构层级模糊
虽然有 `src/layers` 目录，但实际依赖关系混乱：
- `src/core` 依赖 `src/agents`，反之亦然。
- Service (`src/services`) 和 Tool (`src/agents/tools`) 的界限不清，例如 `RAGTool` 内部包含了检索和生成的重逻辑，实际上更像一个 Service。

### 3. 缺乏统一中枢
`UnifiedResearchSystem` 目前承担了过多的职责（路由、执行、错误处理），变成了一个难以扩展的 "上帝类" (God Class)。

## 优化架构的实施价值

你提供的架构图对当前系统有**极高的借鉴和指导价值**，它为我们清理技术债务指明了方向。以下是具体的映射与实施建议：

### 1. 统一入口：Configurable Router
*   **当前**: 多个 Router 混战。
*   **目标**: 废弃 `EntryRouter`, `IntelligentRouter` 等，实现单一的 `ConfigurableRouter`。
*   **价值**: 收敛流量入口，统一路由策略（基于规则 vs 基于模型），便于做 A/B 测试和灰度发布。

### 2. 核心编排：Execution Coordinator
*   **当前**: `UnifiedResearchSystem` + `IntelligentOrchestrator`。
*   **目标**: 提取纯粹的编排逻辑到 `ExecutionCoordinator`。建议使用状态机（如 LangGraph）来管理流程，而不是硬编码的 `if-else`。
*   **价值**: 解耦 "做什么" (Agent) 和 "怎么做" (Workflow)。支持 ReAct、CoT 等多种模式的灵活切换。

### 3. 全链路上下文：Context Manager
*   **当前**: 状态分散在 `ContextManager` 和各种 `*Context*` 类中。
*   **目标**: 合并所有上下文逻辑到一个单例或请求级 `ContextManager`。
*   **价值**: 确保 Agent 间共享一致的 Memory，解决多轮对话中信息丢失的问题。

### 4. 纯粹的 Agent 与 Tool Registry
*   **当前**: Agent 和 Tool 职责不清。`RAGTool` 太重。
*   **目标**: 
    *   **Tool Registry**: 保持现状（`src/agents/tools/tool_registry.py` 设计得不错），强制所有 Agent 通过它获取工具。
    *   **Specialist Agents**: 将 Agent 瘦身，Agent 只负责 "思考" (Prompting & Decision)，脏活累活交给 Tool/Service。
    *   拆解 `RAGTool` 为 `RetrievalTool` (给 Knowledge Agent 用) 和 `GeneratorTool` (给 Answer Agent 用)。

### 5. 基础设施下沉：System Services
*   **当前**: 监控和错误处理散落在各个业务逻辑中。
*   **目标**: 使用装饰器或中间件模式（AOP）集成 `PerformanceMonitor` 和 `ErrorHandler`。
*   **价值**: 业务代码更干净，不再夹杂大量的 `try-except` 和 `log_time`。

## 下一步行动计划 (Action Plan)

我建议分三阶段进行重构，以平滑过渡：

### Phase 1: 瘦身与定义 (Cleaning & Defining)
1.  **定义接口**: 按照新架构图，在 `src/interfaces` 中定义 `IRouter`, `ICoordinator`, `IAgent` 等核心接口。
2.  **清理冗余**: 标记 `Advanced...`, `Intelligent...` 等重复类为 `@deprecated`，梳理出唯一可信的实现类。

### Phase 2: 核心重组 (Core Refactoring)
1.  **实现 Coordinator**: 创建 `src/core/execution_coordinator.py`，接管 `UnifiedResearchSystem` 的编排逻辑。
2.  **重构 Router**: 提取路由逻辑到 `src/core/configurable_router.py`。
3.  **统一 Context**: 合并上下文管理到 `src/core/context_manager.py`。

### Phase 3: 迁移与切换 (Migration)
1.  **Agent 适配**: 修改现有 Agent 继承自新的基类，适配 Coordinator 的调用方式。
2.  **切换入口**: 将 `main.py` 的入口指向新的 `ConfigurableRouter`。

这个计划将把系统从目前的"功能堆砌"状态转变为"清晰分层"的现代化架构。
