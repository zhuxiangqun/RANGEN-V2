# 系统架构优化与重构方案（最终完善版）

基于之前的讨论和您的补充建议，我整合了一份更加详尽、可操作性更强的最终方案。本方案旨在彻底解决"形容词膨胀"问题，建立基于 LangGraph 的现代化 Agent 架构。

## 🎯 核心目标

1. **架构瘦身**: 移除 `Intelligent...`, `Advanced...` 等冗余前缀类，回归单一职责。
2. **编排现代化**: 使用 LangGraph 状态机替代 `if-else` 工作流。
3. **工具原子化**: 将大而全的 `RAGTool` 拆解为可独立复用的原子工具。
4. **无感迁移**: 通过 Shadow Mode (双轨运行) 确保业务零中断。

***

## 📅 分阶段实施计划

### Phase 1: 瘦身与定义 (Cleaning & Defining) - \[预计 1 周]

**目标**: 明确边界，停止出血。

1. **接口定义**:

   * 在 `src/interfaces/architecture_interfaces.py` 中定义 `IRouter`, `ICoordinator`, `IAgent`。(已完成)
2. **标记废弃**:

   * 扫描并标记 `IntelligentRouter`, `AdvancedContextManager` 等类为 `@deprecated`。

   * 创建 `docs/architecture/COMPONENT_MAPPING.md` 作为迁移指南。

### Phase 2: 核心重组 (Core Refactoring) - \[预计 2-3 周]

**目标**: 构建新内核。

1. **Coordinator 实现 (LangGraph)**:

   * 创建 `src/core/coordinator/execution_coordinator.py`。

   * 定义 `AgentState` (messages, current\_step, tools\_output)。

   * 构建 `Reason -> Act -> Observe` 循环图。
2. **Agent 基类重构**:

   * 创建 `src/agents/base/modern_agent.py`。

   * 实现 `execute(task, context)` 标准接口。
3. **工具拆分**:

   * 将 `RAGTool` 拆解为 `RetrievalTool`, `RerankTool`, `GenerationTool`。

   * 更新 `ToolRegistry`。
4. **统一 Context**:

   * 实现 `src/core/context/unified_context.py`，合并分散的上下文逻辑。

### Phase 3: 迁移与切换 (Migration & Switch) - \[预计 1-2 周]

**目标**: 平滑过渡。

1. **MigrationProxy**:

   * 实现 `src/core/migration_proxy.py`，支持同时调用新旧 Router。
2. **双轨运行 (Shadow Mode)**:

   * 配置 `ENABLE_SHADOW_MODE=true`。

   * 记录新旧系统的输出差异（Result, Latency, Success Rate）。
3. **正式切换**:

   * 更新入口 `main.py` 指向 `ConfigurableRouter`。

   * 移除旧代码（在稳定运行 2 周后）。

***

## 🛡️ 风险控制

1. **数据一致性**: 新旧系统共享同一个底层的 Knowledge Base 和 Vector DB，确保检索源一致。
2. **性能回退**: 在 Phase 2 早期引入基准测试，确保 LangGraph 引入的 overhead 可控。
3. **兼容性**: 保持 API 接口（输入输出格式）不变，对前端透明。
