# 深度诊断与修复计划：解决向量检索相似度异常 (Score=0.0)

经过对代码库的深入分析，我发现 `VectorIndexBuilder.search` 方法中存在逻辑缺陷，可能导致在某些情况下返回的相似度为 0 或结果不准确。此外，`TextProcessor` 的实现也可能与向量维度不匹配有关。

## 问题根源分析

1.  **相似度计算逻辑**：
    *   在 `src/knowledge_management_system/core/vector_index_builder.py` 中，使用的是 `faiss.IndexFlatIP` (内积)。对于归一化向量，内积等于余弦相似度。
    *   **关键问题**：如果查询向量或索引向量未能正确归一化，或者存在全零向量，内积计算结果可能异常。
    *   **代码逻辑**：`search` 方法中有归一化操作 `query_vector = query_vector / np.linalg.norm(query_vector)`，但如果 `norm` 为 0（全零向量），这里会产生 NaN 或错误。

2.  **阈值过滤逻辑**：
    *   代码中有一处逻辑：`if similarity_threshold > 0.0 and similarity < similarity_threshold: continue`。
    *   当相似度计算异常（如 0.0）且默认阈值设置不当时，可能导致有效结果被过滤，或者异常结果被错误返回。

3.  **向量维度一致性**：
    *   `TextProcessor` 默认维度是 768，但也支持本地模型动态获取维度。如果构建索引时使用的维度与查询时使用的模型维度不一致，会导致严重错误。

## 执行计划

我将分三步进行诊断和修复：

### 第一步：诊断向量索引状态 (Diagnostic Script)
编写一个独立的诊断脚本 `diagnose_vector_index.py`，执行以下检查：
1.  **加载索引**：检查 FAISS 索引文件是否能成功加载，读取 `ntotal` (总向量数)。
2.  **向量抽样检查**：从索引中重建（reconstruct）几个向量，检查是否存在全零向量或非归一化向量。
3.  **维度一致性检查**：对比索引维度与当前 `TextProcessor` 输出的维度是否一致。
4.  **模型验证**：运行一次简单的 `TextProcessor.encode("test")`，确保输出向量非空且维度正确。

### 第二步：修复 `VectorIndexBuilder` (Core Fix)
根据诊断结果，修正 `src/knowledge_management_system/core/vector_index_builder.py`：
1.  **安全归一化**：在 `add_vector` 和 `search` 中增加对零向量的检查，避免除以零错误。
2.  **日志增强**：在搜索返回 0 结果时，增加 debug 日志输出查询向量的模长和部分原始值。

### 第三步：验证修复效果
重新运行 `verify_kb_content.py`，确认检索结果的相似度得分恢复正常（应 > 0.0）。

## 预期结果
*   确定是索引数据问题（需重建）还是检索逻辑问题（需代码修复）。
*   恢复向量检索的准确性，确保 Top1 结果具有合理的相似度得分。
