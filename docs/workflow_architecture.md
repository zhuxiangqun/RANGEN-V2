# RANGEN 工作流架构设计

## 核心理念

```
基盘 = 执行引擎 + 内部工作流
用户工作流 = 数据 (由用户或自动创建系统生成)
```

---

## 一、执行引擎

**文件**: `src/core/production_workflow.py`

基于 langgraph_unified_workflow.py 简化而来的生产版本。

### 对比

| 指标 | langgraph_unified_workflow.py | production_workflow.py |
|------|-------------------------------|----------------------|
| 行数 | 3232 | ~350 |
| 状态字段 | 60+ | 30 |
| 节点数 | 15+ | 6 |
| RAG/CE/PE | ✅ | ✅ |
| 错误处理 | ✅ | ✅ |

### 核心节点

```
route_query → [simple/complex/reasoning] → synthesize → format → END
```

| 节点 | 功能 |
|------|------|
| route_query | 路由判断 |
| simple_query | 简单查询 |
| complex_query | 复杂查询 + RAG |
| reasoning | 深度推理 + PE/CE |
| synthesize | 综合结果 |
| format | 格式化输出 |

## 二、内部工作流 (Platform Internal)

| 工作流 | 作用 |
|--------|------|
| **启动工作流** | 初始化组件 |
| **健康检查** | 验证状态 |
| **治理工作流** | 监控优化 |

## 三、用户工作流 (数据)

| 类型 | 存储位置 | 说明 |
|------|---------|------|
| **自动创建** | `data/auto_teams.json` | 用户输入需求后自动生成 |
| **用户自定义** | `data/user_teams.json` | 用户手动创建 |
| **模板** | `data/workflow_templates.json` | 预定义模板 |

## 五、API 接口

```python
# 用户工作流
POST /api/v1/workflow/execute     # 执行工作流
GET  /api/v1/workflow/list      # 列出用户工作流

# 内部工作流
GET  /api/v1/system/health      # 健康检查
POST /api/v1/system/startup      # 启动系统
POST /api/v1/system/governance   # 运行治理
```

## 七、总结

| 类别 | 数量 | 说明 |
|------|------|------|
| **执行引擎** | 1个 | 统一入口 |
| **内部工作流** | 3个 | 启动/健康/治理 |
| **用户工作流** | N个 | 存储为数据 |

**核心原则**: 
- 基盘只有 1 个执行引擎 + 3 个内部工作流
- 用户工作流是数据，由用户或自动创建系统生成
- 清晰分离"平台"和"交付物"
