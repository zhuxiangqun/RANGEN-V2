# 测试超时分析指南

## 问题描述

测试执行时间超过预期，导致超时失败。例如：测试1（持久化检查点）执行了710.70秒，超过了700秒的超时限制。

## 分析工具

已创建性能分析工具 `tests/performance_analyzer.py`，用于记录和分析测试执行过程中每个操作的耗时。

## 使用方法

### 1. 运行测试并查看性能摘要

运行测试时，性能分析器会自动记录每个操作的执行时间：

```bash
python tests/run_single_test.py 1
```

测试完成或超时时，会自动打印性能摘要，显示：
- 每个操作的总耗时
- 每个操作的执行次数
- 每个操作的平均耗时、最大耗时、最小耗时
- 每个操作占总时间的百分比

### 2. 分析性能摘要

查看性能摘要，重点关注：

#### a) 总耗时最长的操作
找出总耗时最长的操作，这些操作很可能导致超时。

**示例**：
```
操作名称                                              总耗时      占比
第一次查询执行（保存检查点）                         565.83秒    79.6%
从检查点恢复执行                                     120.45秒    16.9%
```

**分析**：第一次查询执行耗时565.83秒，占总时间的79.6%，是导致超时的主要原因。

#### b) 重复执行的操作
查看执行次数大于1的操作，重复执行可能导致总耗时增加。

**示例**：
```
操作名称                                              次数      总耗时
API调用                                               45        120.50秒
知识检索                                             12        45.30秒
```

**分析**：API调用执行了45次，总耗时120.50秒，平均每次2.68秒。如果减少API调用次数，可以显著降低总耗时。

#### c) 不稳定的操作
比较平均耗时和最大耗时，如果差异很大，说明操作执行时间不稳定。

**示例**：
```
操作名称                                              平均耗时    最大耗时
知识检索                                             3.78秒     15.23秒
```

**分析**：知识检索平均耗时3.78秒，但最大耗时15.23秒，说明某些情况下知识检索非常慢，需要优化。

### 3. 深入分析具体操作

如果某个操作耗时很长，可以：

1. **查看日志**：检查该操作执行时的详细日志，找出耗时原因
2. **检查网络**：如果是API调用，检查网络连接和API服务状态
3. **检查数据量**：如果是数据处理，检查数据量是否过大
4. **检查算法**：如果是计算操作，检查算法复杂度

## 常见超时原因

### 1. API调用过多或过慢

**症状**：
- 性能摘要中API调用总耗时很长
- API调用次数很多

**解决方案**：
- 检查API服务状态
- 优化API调用逻辑，减少不必要的调用
- 使用缓存减少重复调用
- 增加API超时时间（如果合理）

### 2. 知识检索耗时过长

**症状**：
- 知识检索操作耗时很长
- 知识检索执行次数很多

**解决方案**：
- 优化知识检索算法
- 减少检索范围
- 使用更快的检索方法
- 缓存检索结果

### 3. 推理步骤过多

**症状**：
- 推理步骤数量很多
- 每个推理步骤都耗时较长

**解决方案**：
- 优化查询复杂度判断，避免将简单查询识别为复杂查询
- 减少推理步骤数量
- 优化推理算法

### 4. 死锁或无限循环

**症状**：
- 某个操作一直不结束
- 性能摘要中某个操作耗时异常长

**解决方案**：
- 检查代码逻辑，找出死锁或无限循环
- 添加超时机制
- 使用性能分析器定位具体位置

## 优化建议

### 1. 减少不必要的操作

通过性能摘要找出重复或不必要的操作，优化代码逻辑。

### 2. 并行化操作

如果某些操作可以并行执行，使用 `asyncio.gather` 并行执行。

### 3. 使用缓存

对于重复的操作，使用缓存避免重复执行。

### 4. 优化算法

对于耗时的算法，考虑使用更高效的算法。

## 示例分析

### 场景：测试1超时（710.70秒）

**性能摘要**：
```
操作名称                                              总耗时      占比
第一次查询执行（保存检查点）                         565.83秒    79.6%
从检查点恢复执行                                     120.45秒    16.9%
创建第一个工作流实例                                 15.23秒    2.1%
创建第二个工作流实例（模拟重启进程）                 8.67秒     1.2%
```

**分析**：
1. 第一次查询执行耗时565.83秒，占总时间的79.6%，是主要瓶颈
2. 从检查点恢复执行耗时120.45秒，占总时间的16.9%，也需要优化

**优化方向**：
1. 优化第一次查询执行：检查为什么查询"Test"需要这么长时间
   - 是否被错误识别为复杂查询？
   - 是否执行了过多的推理步骤？
   - 是否进行了过多的API调用？
2. 优化检查点恢复：检查为什么恢复需要这么长时间
   - 是否加载了过多的状态？
   - 是否重新执行了不必要的操作？
