# 测试1超时原因分析

## 性能摘要

从性能分析摘要可以看出：

```
操作名称                                              总耗时      占比
第一次查询执行（保存检查点）                         456.64秒    64.9%
从检查点恢复执行                                     246.51秒    35.0%
创建第一个工作流实例                                 0.77秒      0.1%
创建第二个工作流实例（模拟重启进程）                 0.12秒      0.0%
```

**总计：703.15秒，超过700秒超时限制**

## 问题分析

### 1. 第一次查询执行耗时过长（456.64秒）

从日志分析，第一次查询执行"Test"时：

1. **查询被识别为simple查询**（复杂度评分: 1.00）
2. **但仍然执行了完整的推理流程**：
   - 推理步骤生成：超时60秒，然后使用fallback方法
   - 证据检索：检索到5条知识
   - 答案提取：多个API调用
   - 最终答案合成：42.07秒的API调用

3. **主要耗时操作**：
   - 推理步骤生成超时：60秒（等待deepseek-reasoner响应）
   - 最终答案合成：42.07秒（deepseek-reasoner响应）
   - 多个API调用：每个1-17秒不等
   - 模型加载：spaCy和Sentence-BERT模型加载（约3.7秒）

### 2. 从检查点恢复执行耗时过长（246.51秒）

从检查点恢复后，仍然执行了完整的推理流程，包括：
- 重新初始化推理引擎
- 重新执行推理步骤
- 多个API调用

### 3. 测试超时后还在调用API

**原因**：
- 测试超时后，虽然取消了`test_task`，但工作流内部的异步操作（如HTTP请求）可能还在继续执行
- `asyncio.Task.cancel()`只会取消任务本身，但任务内部的子任务（如HTTP请求）可能没有被正确取消
- HTTP连接池没有被立即关闭

**解决方案**：
- 超时后立即关闭所有HTTP连接池
- 取消所有待处理的asyncio任务
- 等待任务取消完成（最多1秒）

## 优化建议

### 1. 优化查询执行路径

对于simple查询（如"Test"），应该：
- 跳过推理步骤生成，直接使用知识检索
- 减少不必要的API调用
- 使用更快的答案生成路径

### 2. 优化检查点恢复

从检查点恢复时，应该：
- 避免重新初始化推理引擎
- 直接使用已保存的推理结果
- 跳过重复的推理步骤

### 3. 增加测试超时时间

考虑到：
- 第一次查询执行：456.64秒
- 从检查点恢复执行：246.51秒
- 总计：703.15秒

建议将测试超时时间增加到**800秒**（约13.3分钟），以提供足够的缓冲时间。

### 4. 优化测试查询

使用更简单的查询，或者：
- 使用mock数据避免实际API调用
- 使用缓存避免重复的推理步骤生成
- 减少ChiefAgent迭代次数（已启用FAST_TEST_MODE）
