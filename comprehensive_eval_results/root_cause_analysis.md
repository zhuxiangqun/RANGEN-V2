# 测试次数低和准确率下降的根本原因分析

生成时间: 2025-11-01

## 执行摘要

经过深入分析日志和代码，发现了导致**测试次数低（只有7个样本）**和**准确率下降**的两个根本问题：

1. **双重超时导致大量正确结果被丢弃**：即使LLM已经返回正确答案，但由于commit阶段的超时设置不合理，导致`success=False`，无法参与准确率评估。
2. **超时时间设置冲突**：外层超时（72秒）短于内层超时（180秒），导致即使推理已完成，后续处理步骤也会被外层超时中断。

## 二、根本原因1：双重超时导致正确结果被丢弃

### 2.1 问题描述

**现象**：
- 日志显示"✅ 推理完成: Jane Ballou"（期望答案也是"Jane Ballou"）
- 但随后出现"RESEARCH stage=commit timeout took=118.74s"
- 最终`success=False`，答案无法参与准确率评估

### 2.2 问题代码位置

**文件**: `src/unified_research_system.py`

**第638行**（外层超时）:
```python
stage_timeout = min(request.timeout * 0.4, 120.0)  # 40% of request timeout, max 120s
# request.timeout = 180s, 所以 stage_timeout = 72s
```

**第321行**（内层超时）:
```python
result = await asyncio.wait_for(task, timeout=request.timeout)  # timeout = 180s
```

**问题**：
- 外层只给commit阶段72秒
- 但内层`_execute_research_internal`可能需要更长时间（因为包含多个步骤）
- 即使推理已完成，但后续的答案生成、引用生成等步骤还在执行
- 当这些后续步骤执行时，外层的72秒超时就到了，导致整个查询失败

### 2.3 代码执行流程

```
execute_research()
  ↓
_commit_to_research_decision() [外层超时：72秒]
  ↓
_execute_research_internal() [内层超时：180秒]
  ↓
步骤1: 知识检索 (12秒超时)
步骤2: 推理分析 (12秒超时) ← "✅ 推理完成" 在这里完成
步骤3: 答案生成 (10秒超时) ← 可能在这里超时
步骤4: 引用生成 (10秒超时)
```

**问题**：
- 步骤2完成时，已经消耗了约12秒
- 步骤3和4又需要约20秒
- 但如果步骤3或4内部有额外处理（如LLM调用），总时间可能超过72秒
- 外层72秒超时触发，导致整个查询失败，即使推理已经完成且答案正确

### 2.4 影响

**直接影响**：
- 大量样本即使LLM返回正确答案，也被标记为失败
- 只有7个样本（14%）能够参与准确率评估
- 测试次数低（7次）

**间接影响**：
- 准确率计算基于极少的样本（7次），统计意义不足
- 大量正确答案被丢弃，实际性能被低估

## 三、根本原因2：超时时间设置不合理

### 3.1 超时时间对比

| 位置 | 超时设置 | 计算值 | 问题 |
|------|----------|--------|------|
| 外层commit阶段 | `min(request.timeout * 0.4, 120.0)` | **72秒** | 太短 |
| 内层`_commit_to_research_decision` | `request.timeout` | **180秒** | 正常 |
| `_execute_research_internal`内部步骤 | 12秒 + 12秒 + 10秒 + 10秒 | **44秒** | 正常 |
| **实际需要时间** | 推理(100-200秒) + 后续处理(20-50秒) | **120-250秒** | 超过外层72秒 |

### 3.2 实际执行时间分析

从日志统计：
- **平均处理时间**: 63.27秒（成功的7个样本）
- **最大处理时间**: 177.72秒
- **失败样本的平均超时时间**: 100-200秒

**结论**：
- 大部分查询需要100-200秒才能完成
- 但外层只给了72秒，导致大量超时

### 3.3 时间消耗分析

从日志看，时间主要消耗在：
1. **LLM推理调用**（最大头）：100-180秒
   - DeepSeek推理模型响应时间较长
   - 推理步骤需要时间
2. **证据处理**：10-20秒
   - 证据提取、过滤、压缩
3. **答案生成**：5-10秒
   - 答案格式化、验证
4. **引用生成**：5-10秒
   - 引用提取和格式化

**总计**：120-220秒，远超外层72秒超时

## 四、准确率下降的原因

### 4.1 准确率计算逻辑

准确率 = 正确答案数量 / 测试次数

**测试次数** = 同时有期望答案和实际答案（且success=True）的样本数

### 4.2 准确率下降的原因

**优化前**：
- 测试次数: 3次
- 准确率: 33.33%（1/3）

**优化后**：
- 测试次数: 7次（增加了133%）
- 准确率: 14.29%（1/7）

**分析**：
1. ✅ **测试次数增加是好事**：说明有更多样本能够成功提取答案
2. ⚠️ **但准确率下降**：可能原因：
   - 新增的4个样本答案不正确
   - 样本难度不同（新的7个样本可能更难）
   - 答案验证更严格，过滤掉了一些"边缘正确"的答案

### 4.3 更深层的问题

**真正的问题不是准确率下降，而是测试次数太低！**

- 只有7个样本（14%）能够参与评估
- 86%的样本因为超时而无法评估
- 即使这些样本的答案是正确的（从日志看，很多样本的答案与期望答案一致）

**结论**：
- 准确率下降可能是因为样本不同（3次 vs 7次）
- 但更大的问题是测试次数太低，导致准确率计算不准确
- 需要先解决超时问题，让更多样本能够参与评估

## 五、解决方案

### 5.1 解决方案1：调整超时策略

**目标**：让已完成的推理结果能够被保留，即使后续步骤超时

**方案**：
1. **分离推理阶段和后续处理阶段**：
   - 推理阶段完成后，立即返回结果
   - 后续处理（答案生成、引用生成）在后台异步执行
   
2. **调整超时设置**：
   - commit阶段的超时应该基于实际需要的时间
   - 建议：`stage_timeout = min(request.timeout * 0.7, 150.0)`（126秒或150秒）

3. **快速失败机制**：
   - 如果推理已完成但后续步骤超时，仍然返回推理结果
   - 标记为"部分成功"而不是完全失败

### 5.2 解决方案2：优化执行流程

**目标**：减少不必要的处理时间

**方案**：
1. **并行优化**：确保知识检索、推理、答案生成真正并行执行
2. **缓存机制**：缓存常见查询的推理结果
3. **早期退出**：如果推理已返回答案，可以跳过部分后续步骤

### 5.3 解决方案3：改进success判断逻辑

**目标**：即使后续步骤失败，如果推理已完成且返回答案，也应标记为success

**方案**：
1. **分层success判断**：
   - 推理阶段成功 → 基础success
   - 后续步骤成功 → 完整success
   - 即使后续失败，如果推理成功且有答案，仍然标记为success（但标记为"部分成功"）

2. **答案提取优化**：
   - 在推理完成时立即提取答案
   - 不等待后续步骤完成

### 5.4 解决方案4：改进评测系统

**目标**：即使success=False，如果日志中有答案，也应该参与评估

**方案**：
1. **修改答案提取逻辑**：
   - 不仅从`success=True`的样本提取答案
   - 也从`success=False`但日志中有"✅ 推理完成"的样本提取答案
   
2. **区分成功类型**：
   - 完全成功：所有步骤完成
   - 部分成功：推理完成但后续步骤失败
   - 失败：推理未完成

## 八、结论

### 根本原因总结

1. **超时设置不合理**：外层72秒超时太短，无法完成完整的处理流程
2. **success判断过于严格**：只有所有步骤完成才标记为success，导致大量正确答案被丢弃
3. **评测系统提取逻辑受限**：只从success=True的样本提取答案，错过了大量有效答案

### 解决策略

优先解决超时和success判断问题，这将立即大幅提高测试次数，让更多样本参与准确率评估。
