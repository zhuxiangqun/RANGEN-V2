## 🔍 复杂度差异产生的原因

### 1. **判断依据不同**

| 阶段 | 使用的文本 | 长度 | 包含信息 |
|------|-----------|------|---------|
| **复杂度判断** | 原始查询文本 | ~100-500字符 | 仅查询本身 |
| **实际处理** | 完整prompt（查询+证据） | ~20000+字符 | 查询+证据+上下文 |

**差异**：
- 复杂度判断基于**查询文本本身**
- 实际处理基于**查询+证据+上下文**

### 2. **判断标准不同**

#### LLM复杂度判断标准（基于查询文本）

```python
# 判断规则：
- simple: 单步事实查找
- medium: 多跳事实查找（所有步骤都是直接事实查找）
- complex: 需要复杂推理、计算、分析
```

**示例**：
- 查询："If my future wife has the same first name as the 15th first lady..."
- LLM判断：**complex**（因为需要多跳推理和复杂计算）

#### 规则判断标准（基于查询+证据）

```python
# _calculate_task_complexity() 考虑的因素：
1. 查询类型复杂度（30%）
2. 证据数量复杂度（20%）
3. 查询长度复杂度（15%）
4. 查询语义复杂度（25%）
5. 证据质量复杂度（10%）
```

**差异**：
- LLM判断：基于查询文本的语义理解
- 规则判断：基于查询+证据的统计特征

### 3. **数据集特征导致的差异**

#### frames-benchmark数据集特征

所有查询都是**复杂的多跳推理问题**：
- 多约束条件
- 多跳推理
- 复杂计算和比较
- 时间推理

**结果**：
- ✅ LLM判断所有查询为"complex"是**正确的**
- ✅ 因为查询本身确实需要复杂推理

#### 但实际处理时可能不同

即使查询复杂，如果：
- 证据质量高（相关性>0.85）
- 证据数量适中（5-10个）
- 证据直接回答了问题

可能可以使用快速模型处理。

## 🔍 为什么会产生差异？

### 1. **判断时机问题**

**当前流程**：
```
证据收集 → 复杂度判断 → 模型选择
```

**问题**：
- 复杂度判断时证据已收集，但判断时**不使用证据信息**
- 只基于原始查询文本判断，可能不够准确

**建议**：
- 可以考虑在判断时也考虑证据质量
- 或者先判断复杂度，再根据证据质量调整

### 2. **判断依据不完整**

**当前LLM判断**：
- 只使用原始查询文本（前500字符）
- 不包含证据、上下文、处理历史

**问题**：
- 无法考虑证据质量对复杂度的影响
- 无法考虑上下文对复杂度的影响

**示例**：
- 查询："What is the capital of France?" → LLM判断：simple
- 但如果证据质量很差，可能需要复杂推理 → 实际：complex

### 3. **判断标准不一致**

**LLM判断标准**：
- 基于查询文本的语义理解
- 区分"多跳事实查找"（medium）和"复杂推理"（complex）

**规则判断标准**：
- 基于查询+证据的统计特征
- 考虑证据数量、质量、查询长度等

**问题**：
- 两种判断标准可能产生不同结果
- LLM判断优先，但可能不够准确

## 🔍 具体差异场景

### 场景1：查询复杂但证据质量高

**查询**：
```
"If my future wife has the same first name as the 15th first lady of the United States' mother..."
```

**LLM判断**：
- **complex**（因为需要多跳推理）

**实际处理**：
- 如果证据质量高（相关性>0.85），可能可以用快速模型
- 但当前逻辑：LLM判断为complex → 强制使用推理模型

**差异**：
- 判断：complex（基于查询文本）
- 实际：可能可以用快速模型（基于证据质量）

### 场景2：查询简单但证据复杂

**查询**：
```
"What is the capital of France?"
```

**LLM判断**：
- **simple**（单步事实查找）

**实际处理**：
- 如果证据质量差，可能需要复杂推理
- 但当前逻辑：LLM判断为simple → 允许使用快速模型

**差异**：
- 判断：simple（基于查询文本）
- 实际：可能需要复杂推理（基于证据质量）

### 场景3：多跳事实查找 vs 复杂推理

**查询**：
```
"The artist who released 'Father of Asahd' went to the same school as an Olympic diver. How many teams did this diver participate on?"
```

**LLM判断**：
- **medium**（多跳事实查找，所有步骤都是直接事实查找）

**实际处理**：
- 如果所有步骤都是直接事实查找，可以用快速模型
- 但如果需要推理，需要用推理模型

**差异**：
- 判断：medium（基于查询文本的语义理解）
- 实际：取决于证据是否直接回答了问题

## 🎯 结论

### 1. **复杂度差异的根本原因**

1. **判断依据不同**：
   - 复杂度判断：只使用原始查询文本
   - 实际处理：使用查询+证据+上下文

2. **判断标准不同**：
   - LLM判断：基于查询文本的语义理解
   - 规则判断：基于查询+证据的统计特征

3. **判断时机问题**：
   - 复杂度判断时证据已收集，但判断时不使用证据信息

### 2. **当前系统的行为**

- ✅ LLM判断所有frames-benchmark查询为"complex"是**正确的**
- ✅ 因为查询本身确实需要复杂推理
- ⚠️ 但可能有些查询虽然复杂，但证据质量高，可以用快速模型处理

### 3. **改进建议**

1. **在复杂度判断时考虑证据质量**：
   - 如果证据质量高（相关性>0.85），可以降低复杂度评分
   - 如果证据质量差，可以提高复杂度评分

2. **动态调整复杂度**：
   - 先基于查询文本判断初始复杂度
   - 再根据证据质量动态调整

3. **统一判断标准**：
   - 统一LLM判断和规则判断的标准
   - 确保两种方法产生一致的结果
