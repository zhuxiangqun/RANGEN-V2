# 流式 API 和分多次调用 API 方案分析

**分析时间**: 2025-11-16  
**核心问题**: 如何解决 API 响应被截断的问题

---

## 🔍 问题分析

### 当前问题
- API 响应长度被截断到约 2000 字符
- `finish_reason=stop`，`max_tokens=8000`
- 说明不是达到 token 限制，而是响应被截断

### 可能原因
1. API 侧可能有字符数限制（如 2000 字符）
2. 响应在传输过程中被截断
3. 单次提取的数据量过大

---

## 💡 解决方案

### 方案1: 使用流式 API

**DeepSeek API 支持流式输出**：
- 可以通过设置 `stream` 参数为 `true` 来启用
- 流式 API 会逐块生成内容，适用于实时显示

**优点**：
- 可以实时接收响应
- 对于长响应，可以逐步处理

**缺点**：
- 对于知识图谱构建，我们需要完整的 JSON 响应才能解析
- 流式 API 主要用于实时显示，不适合需要完整 JSON 的场景
- 实现复杂度较高

**结论**：
- ❌ **不推荐用于知识图谱构建**
- ✅ 更适合实时对话、文本生成等场景

---

### 方案2: 分多次调用 API（推荐）

**方案**：
1. **先提取实体**：第一次调用 API，只提取实体（5个实体）
2. **再提取关系**：第二次调用 API，基于已提取的实体提取关系（8条关系）

**优点**：
- 每次调用的响应更短，避免截断
- 实现相对简单
- 可以确保每次调用都在 API 限制范围内

**缺点**：
- 需要两次 API 调用，成本和时间增加
- 需要修改现有代码逻辑

**实现建议**：
- 如果当前方案（5个实体、8条关系）仍然出现截断，可以考虑实现此方案

---

### 方案3: 进一步减少提取数量（已实施）

**当前限制**：
- 实体数量：5 个
- 关系数量：8 条

**如果仍然截断**：
- 可以进一步减少到 3 个实体、5 条关系
- 或者只提取实体，不提取关系（关系可以通过其他方式推断）

1. **流式 API**：不推荐用于知识图谱构建
2. **分多次调用 API**：如果当前方案仍然出现截断，可以考虑实现
3. **当前方案**：已实施减少提取数量，应该能够解决大部分截断问题
