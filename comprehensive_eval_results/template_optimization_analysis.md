# 提示词模板优化分析

## 当前状况

### 模板统计
- **总模板数**: 35个
- **主模板**: 26个
- **Fallback模板**: 9个

### 模板分类
- **classification** (分类): 8个
- **reasoning** (推理): 8个
- **validation** (验证): 5个
- **evidence** (证据): 4个
- **extraction** (提取): 3个
- **detection** (检测): 2个
- 其他: 5个

## 为什么需要这么多模板？

### 1. 多层Fallback机制
系统采用了三层fallback策略：
```
主模板 (完整版)
  ↓ (失败)
Fallback模板 (简化版)
  ↓ (失败)
硬编码Fallback (最小版)
```

**原因**：
- 确保系统在任何情况下都能工作
- 如果主模板不存在或失败，有备用方案
- 避免系统崩溃

### 2. 不同场景的需求
- **主模板**：功能完整，包含详细指导
- **Fallback模板**：简化版，确保基本功能
- **硬编码Fallback**：紧急情况下的最后保障

### 3. 历史演进
- 系统在重构过程中，为了保持向后兼容，保留了旧模板
- 新功能添加时，创建了新模板
- 没有及时清理冗余模板

## 问题分析

### 1. 模板冗余
- 大部分fallback模板只是主模板的简化版
- 某些fallback模板可以动态生成，不需要单独存储
- 某些功能相似的模板可以合并

### 2. 维护成本高
- 35个模板需要分别维护
- 修改一个功能可能需要修改多个模板
- 容易产生不一致

### 3. 查找效率低
- 模板数量多，查找和选择逻辑复杂
- 代码中有大量 `if not prompt: try fallback` 的逻辑

## 优化建议

### 方案1: 统一Fallback生成器（推荐）

**核心思想**：不存储fallback模板，而是动态生成

```python
def generate_fallback_prompt(template_name: str, **kwargs) -> str:
    """根据主模板名称动态生成fallback提示词"""
    # 根据模板类型生成对应的简化版
    if "reasoning" in template_name:
        return generate_simple_reasoning_prompt(**kwargs)
    elif "validation" in template_name:
        return generate_simple_validation_prompt(**kwargs)
    # ...
```

**优势**：
- 减少模板数量（从35个减少到约20个）
- 统一管理fallback逻辑
- 更容易维护

### 方案2: 模板合并

**合并策略**：
1. **相似功能合并**：
   - `answer_extraction` 和 `answer_extraction_llm` → 合并为一个
   - `reasoning_with_evidence` 和 `fallback_reasoning_with_evidence` → 使用动态生成

2. **保留核心模板**：
   - 保留核心功能模板（约15-20个）
   - 移除冗余的fallback模板

### 方案3: 模板分层

**三层结构**：
1. **核心模板**（必需）：约10-15个核心功能模板
2. **扩展模板**（可选）：按需加载的扩展功能模板
3. **动态Fallback**（运行时生成）：不存储，动态生成

## 实施建议

### 短期（立即实施）
1. ✅ **已完成**：添加所有缺失的模板（确保系统正常运行）
2. 🔄 **进行中**：分析哪些模板是真正必需的

### 中期（1-2周）
1. 实现统一Fallback生成器
2. 移除冗余的fallback模板
3. 合并相似功能的模板

### 长期（1个月）
1. 重构模板系统，采用分层架构
2. 实现模板的动态加载和缓存
3. 建立模板使用统计，优化模板选择

## 结论

**当前模板数量多的原因**：
1. 多层fallback机制（主模板 + fallback模板 + 硬编码）
2. 不同场景的需求（完整版 + 简化版）
3. 历史演进（向后兼容 + 新功能添加）

**优化方向**：
1. 使用动态fallback生成器，减少存储的模板数量
2. 合并相似功能的模板
3. 采用模板分层架构，按需加载

**预期效果**：
- 模板数量：35个 → 约20个（减少43%）
- 维护成本：降低
- 系统稳定性：保持或提升
