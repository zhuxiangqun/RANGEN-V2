## 📊 缓存 vs 学习数据的区别

### 1. 缓存（LLM Cache）

**作用**: 保存LLM调用的结果，避免重复调用相同的LLM请求

**内容**:
```json
{
  "cache_key_hash": {
    "result": "LLM的响应内容",
    "timestamp": 1234567890
  }
}
```

**缓存键生成**:
- 基于 `func_name` + `prompt` 内容
- 提取查询内容作为缓存键的核心
- 使用MD5哈希生成唯一键

**适用场景**:
- ✅ **同一运行内**的重复调用（如多次验证相同答案）
- ✅ **相同查询**的重复调用（如测试时重复运行相同样本）
- ❌ **不同查询**的跨运行调用（不会命中）

**问题**:
- 如果每次运行的查询都不同，缓存不会命中
- 缓存命中率低可能是**正常的**（如果查询不同）

### 2. 学习数据（Learning Data）

**作用**: 保存历史经验，用于改进系统配置和策略

**内容**:
```json
{
  "error_patterns": {
    "query_type": ["错误模式1", "错误模式2"]
  },
  "success_patterns": {
    "query_type": {
      "count": 10,
      "avg_confidence": 0.85
    }
  },
  "performance_metrics": {
    "query_type": {
      "avg_time": 120.5,
      "success_rate": 0.95
    }
  },
  "adaptive_weights": {
    "query_type": 1.2
  },
  "query_difficulty_scores": {
    "query_type": 0.7
  }
}
```

**适用场景**:
- ✅ **跨运行**应用历史经验
- ✅ **不同查询**也能应用（基于查询类型、模式等）
- ✅ 根据历史经验调整系统配置和策略

**应用方式**:
- 在 `reason()` 方法开始时调用 `apply_learned_insights()`
- 根据历史经验调整：
  - 置信度阈值
  - 模型选择策略
  - 证据处理策略
  - 推理步骤数量

## 🔍 第二次运行应该起作用的

### ✅ 学习数据（Learning Data）

**应该起作用**:
1. **性能指标**: 根据历史性能调整配置
2. **错误模式**: 避免重复历史错误
3. **成功模式**: 复用历史成功策略
4. **自适应权重**: 调整不同查询类型的处理权重
5. **查询难度评分**: 根据历史难度调整处理策略

**预期效果**:
- 系统配置更合理（基于历史经验）
- 处理策略更优化（避免已知错误）
- 性能指标可能提升（基于历史优化）

### ❌ 缓存（LLM Cache）

**不应该起作用**（如果查询不同）:
- 缓存只对**相同查询**有效
- 如果第二次运行的是不同查询，缓存不会命中
- 缓存命中率低是**正常的**

**可能起作用**（如果查询相同）:
- 如果测试时重复运行相同样本，缓存会命中
- 同一运行内的重复调用会命中缓存
