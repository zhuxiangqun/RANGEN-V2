## 🔍 性能瓶颈分析

### 瓶颈1: Wikipedia内容抓取（主要瓶颈）⏱️

**代码位置**: `wikipedia_fetcher.py`

**耗时因素**:
1. **请求延迟**: `self.request_delay = 0.5` 秒（每个请求之间）
2. **网络延迟**: 每个Wikipedia API请求需要1-3秒
3. **重试机制**: 失败时最多重试3次，每次延迟2秒
4. **完整文本抓取**: `include_full_text=True`，需要抓取完整HTML并解析

**计算**:
- 每个item平均有2-5个Wikipedia链接
- 每个链接: 0.5秒延迟 + 2-3秒请求 = 2.5-3.5秒
- 5个链接: 5 × 3秒 = 15秒
- 加上重试和错误处理: **20-30秒/item**

### 瓶颈2: 向量化（次要瓶颈）⏱️

**代码位置**: `service_interface.py` Line 606

**耗时因素**:
1. **Jina API调用**: 每个向量化请求需要1-2秒
2. **长文本分块**: 长文本需要分块向量化（11个块）
3. **网络延迟**: API调用有网络延迟
4. **批量处理**: 虽然批量导入，但向量化是串行的

**从日志看**:
- 每个条目需要11个向量化块
- 每个块约1秒
- 总向量化时间: **11-15秒/item**

### 瓶颈3: 内容清理（较小瓶颈）⏱️

**代码位置**: `wikipedia_fetcher.py` `_extract_text_from_html`

**耗时因素**:
1. **BeautifulSoup解析**: HTML解析需要0.5-1秒
2. **正则表达式清理**: 多次正则匹配需要0.1-0.2秒
3. **文本格式化**: 文本处理需要0.1-0.2秒

**总清理时间**: **0.5-1秒/item**

## 💡 优化建议

### 优化1: 减少Wikipedia请求延迟（优先级：高）

**当前**: `request_delay = 0.5` 秒

**建议**: 
- 对于批量导入，可以降低到0.2-0.3秒
- 或者使用并发请求（但要注意Wikipedia的速率限制）

**预期效果**: 减少30-40%的抓取时间

### 优化2: 并行化Wikipedia抓取（优先级：中）

**当前**: 串行抓取，一个接一个

**建议**: 
- 使用异步并发（asyncio）同时抓取多个页面
- 限制并发数（如5-10个并发）

**预期效果**: 减少50-70%的抓取时间

### 优化3: 优化向量化策略（优先级：中）

**当前**: 每个条目串行向量化

**建议**:
- 批量向量化（如果API支持）
- 或者使用本地embedding模型（避免API延迟）

**预期效果**: 减少20-30%的向量化时间

### 优化4: 增加批处理大小（优先级：低）

**当前**: `batch_size = 5`

**建议**: 增加到10-20

**注意**: 需要更多内存，但可以减少总处理时间

**预期效果**: 减少10-15%的总时间
