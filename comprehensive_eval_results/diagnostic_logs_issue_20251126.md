## 💡 可能的原因

### 原因1: 日志级别设置问题

**问题**: 诊断日志使用`logger.warning`级别，但日志系统可能只记录INFO或更高级别的日志

**检查方法**:
- 检查日志系统配置
- 检查日志级别设置
- 确认WARNING级别的日志是否被记录

---

### 原因2: 日志系统使用自定义记录器

**问题**: 系统可能使用自定义的日志记录器（如`log_info`），而不是标准的`logger.warning`

**证据**:
- 代码中使用了`log_info`函数
- 日志格式不包含日志级别标记
- 日志内容直接输出，没有格式

**检查方法**:
- 检查`src/utils/research_logger.py`
- 确认日志系统如何配置
- 确认是否需要使用`log_info`而不是`logger.warning`

---

### 原因3: 代码路径未执行

**问题**: 模型选择逻辑可能没有被执行

**证据**:
- 所有样本都使用了推理模型
- 未找到模型选择相关的日志
- 但测试正常运行，说明代码在执行

**检查方法**:
- 检查模型选择逻辑是否在正确的代码路径中
- 确认`_select_llm_for_task`是否被调用
- 检查是否有其他日志输出

---

## 🔧 解决方案

### 方案1: 使用log_info记录诊断日志

**如果日志系统使用自定义记录器，应该使用`log_info`而不是`logger.warning`**:

```python
from src.utils.research_logger import log_info

log_info(f"🔍 [诊断] [模型选择] 开始选择模型...")
```

---

### 方案2: 检查日志级别设置

**检查日志系统配置，确认日志级别设置**:

```python
# 检查日志级别
import logging
print(f"日志级别: {logging.getLogger().level}")
print(f"根日志级别: {logging.root.level}")
```

---

### 方案3: 同时使用两种日志方式

**同时使用`logger.warning`和`log_info`，确保日志被记录**:

```python
self.logger.warning(f"🔍 [诊断] [模型选择] 开始选择模型...")
log_info(f"🔍 [诊断] [模型选择] 开始选择模型...")
```
