# 批量LLM调用截断保护措施

**实施时间**: 2025-11-16  
**状态**: ✅ 已完成

---

## 问题分析

批量LLM调用确实可能导致响应被截断，原因：
1. **响应更长**：批量处理将多个文本的结果组合在一起
   - 5个文本 × 2个实体 = 最多10个实体
   - 5个文本 × 3条关系 = 最多15条关系
   - 响应长度可能超过API限制

2. **API侧字符数限制**：即使设置了`max_tokens=8192`，API仍然可能在200或2000字符处截断响应

---

## 实施的保护措施

### 1. 减少批量大小 ✅

**修改位置**: `knowledge_management_system/api/service_interface.py` (line 1797)

**改进**:
- 将批量大小从5个文本减少到3个文本
- 降低响应长度，减少截断风险
- 3个文本 × 2个实体 = 最多6个实体（比10个更安全）

**预期效果**:
- 响应长度减少约40%
- 截断风险显著降低

---

### 2. 截断检测 ✅

**修改位置**: `knowledge_management_system/api/service_interface.py` (lines 1877-1899, 2046-2071)

**检测逻辑**:
```python
# 检查是否疑似被截断（长度正好是200或2000的倍数）
is_suspicious_length = (
    response_length > 0 and 
    (response_length == 200 or response_length == 2000 or 
     (response_length % 200 == 0 and response_length < 3000) or
     (response_length % 1000 == 0 and response_length < 5000))
)
```

**检测时机**:
- 批量实体提取响应后
- 批量关系提取响应后

**处理方式**:
- 如果检测到疑似截断，立即回退到单个处理
- 确保每个条目都能正常处理

---

### 4. 多层回退机制 ✅

**回退策略**:
1. **第一层**：截断检测 → 回退到单个处理
2. **第二层**：JSON解析失败 → 尝试部分解析
3. **第三层**：部分解析失败 → 正则提取
4. **第四层**：正则提取失败 → 回退到单个处理

**确保**:
- 即使批量处理失败，也能正常处理每个条目

## 性能影响

### 批量大小调整

**原始**:
- 批量大小：5个文本
- 每批API调用：2次（批量提取实体 + 批量提取关系）
- 18265条数据：约3653批 → 7306次API调用

**调整后**:
- 批量大小：3个文本
- 每批API调用：2次（批量提取实体 + 批量提取关系）
- 18265条数据：约6088批 → 12176次API调用

**影响**:
- API调用次数增加约67%（从7306次增加到12176次）
- 但仍然比单个处理（36530次）减少约67%
- 截断风险显著降低

---

## 权衡分析

### 选项1：批量大小=5（原始）
- ✅ API调用次数更少（7306次）
- ❌ 截断风险较高（响应更长）

### 选项2：批量大小=3（当前）
- ✅ 截断风险较低（响应更短）
- ✅ 仍然比单个处理减少67%的API调用
- ⚠️ API调用次数比选项1多67%

### 选项3：批量大小=2
- ✅ 截断风险最低
- ⚠️ API调用次数：18265条 → 约9133批 → 18266次API调用
- ⚠️ 性能提升有限（只减少50%）

---

## 建议

**当前配置（批量大小=3）是合理的权衡**：
1. ✅ 显著降低截断风险
2. ✅ 仍然大幅减少API调用次数（67%）
3. ✅ 有完善的截断检测和回退机制

**如果仍然遇到截断问题**：
- 可以进一步减少批量大小到2
- 或者增加截断检测的敏感度
