## 🔍 根本原因分析

### 1. 两阶段流水线效率问题 ⚠️ **主要瓶颈**

**代码位置**: `src/core/real_reasoning_engine.py:10157-10332`

**问题**:
1. **快速模型尝试失败后的fallback**:
   - 快速模型尝试：3-10秒
   - 质量检查失败（答案提取失败或置信度低）
   - Fallback到推理模型：200-240秒
   - **总耗时 = 快速模型时间 + 推理模型时间 = 203-250秒**

2. **复杂查询判断不够严格**:
   ```python
   # 当前逻辑：复杂度>=7才跳过快速模型
   is_complex_query = (
       isinstance(dynamic_complexity, (int, float)) and dynamic_complexity >= 7
   ) or (
       llm_complexity_for_pipeline == 'complex'
   )
   ```
   - **问题**: 复杂度6的查询仍然会尝试快速模型，可能失败后fallback
   - **建议**: 将阈值降低到6，或更严格地判断

3. **质量检查可能过于严格**:
   - 答案提取失败 → fallback
   - 置信度低 → fallback
   - **问题**: 某些情况下，快速模型的答案可能已经足够好，但被误判为需要fallback

**优化方案**:
1. **更严格的复杂度判断**:
   - 复杂度>=6直接使用推理模型
   - 或使用更智能的判断（基于查询类型、证据数量等）

2. **优化质量检查**:
   - 降低置信度阈值（对于简单查询）
   - 改进答案提取逻辑（支持更多格式）

3. **快速失败机制**:
   - 如果快速模型响应时间过长（>15秒），直接fallback
   - 避免等待快速模型超时
