# Fallback答案提取优化报告（2025-11-23）

## 问题描述

从运行日志（样本20）中发现，fallback逻辑提取的答案质量很差：

```
⚠️ Fallback提取的答案基础检查失败 | 答案: doi | 尝试次数: 1/3
⚠️ Fallback提取的答案基础检查失败 | 答案: 3.4 mi | 尝试次数: 2/3
⚠️ 前2次Fallback尝试都失败，提前退出，避免第3次调用
```

**问题**：
- 提取的答案"doi"明显是元数据字段（可能是从"DOI: xxx"中提取的）
- 提取的答案"3.4 mi"是单位缩写（可能是从"3.4 miles"中提取的，但只提取了部分）
- 这些无效答案通过了初步提取，但被基础检查拒绝，浪费了fallback尝试次数

## 根本原因

`_extract_answer_simple`方法的过滤规则不够严格：

1. **冒号提取模式太宽泛**：
   - 会提取所有冒号后的内容，包括"DOI: xxx"、"URL: xxx"等元数据
   - 没有检查冒号前的字段名是否为元数据字段

2. **缺少元数据字段过滤**：
   - 没有排除常见的元数据字段（DOI、URL、页码等）
   - 没有排除单位缩写（mi、km等）

3. **数字提取不够精确**：
   - 对于"3.4 mi"这样的内容，可能只提取了数字部分或单位部分

## 修复方案

### 1. 增强冒号提取模式的过滤 ✅

**文件**: `src/core/real_reasoning_engine.py`

**修改**:
- 添加元数据前缀检查
- 排除只包含数字和分隔符的短字符串（可能是页码、DOI等）
- 排除以元数据前缀开头的候选答案
- 排除单位缩写单独出现的情况

**代码**:
```python
# 🚀 P0修复：排除元数据字段前缀
metadata_prefixes = [
    'doi', 'url', 'http', 'https', 'www', 'page', 'pages', 'pp', 'vol', 'volume',
    'issn', 'isbn', 'pmid', 'arxiv', 'ref', 'reference', 'fig', 'figure', 'table',
    'chapter', 'section', 'para', 'paragraph', 'line', 'lines', 'p.', 'pp.',
    '页码', '页', '卷', '期', '编号', '链接', '网址'
]

# 排除只包含数字和常见分隔符的短字符串
if len(candidate) <= 20 and re.match(r'^[\d\s\.\-\/\\]+$', candidate):
    continue

# 排除单位缩写
unit_abbreviations = ['mi', 'km', 'm', 'cm', 'mm', 'ft', 'in', 'yd', 'lb', 'kg', 'g', 'mg']
if candidate_lower.strip() in unit_abbreviations:
    continue
```

### 2. 增强括号提取模式的过滤 ✅

**修改**:
- 添加元数据标记检查（包括DOI、URL等）
- 排除只包含数字和分隔符的元数据

### 3. 增强行提取模式的过滤 ✅

**修改**:
- 扩展元数据模式列表（包括DOI、URL、页码等）
- 排除以元数据字段开头的行
- 排除只包含数字和单位缩写的短行

## 修复效果

### 预期效果

1. **避免提取元数据字段**：
   - "DOI: xxx" → 不会提取"doi"
   - "URL: xxx" → 不会提取URL
   - "Page 123" → 不会提取页码

2. **避免提取单位缩写**：
   - "3.4 mi" → 不会单独提取"mi"
   - 完整的数字+单位组合会被正确处理

3. **提高fallback成功率**：
   - 减少无效答案的提取
   - 提高有效答案的提取概率
   - 减少fallback尝试次数

### 过滤规则

**排除的元数据字段**：
- DOI、URL、HTTP、HTTPS、WWW
- 页码、卷号、期号
- 引用、图表编号
- 章节、段落编号

**排除的单位缩写**：
- 长度单位：mi、km、m、cm、mm、ft、in、yd
- 重量单位：lb、kg、g、mg

**排除的模式**：
- 只包含数字和分隔符的短字符串（≤20字符）
- 以元数据字段开头的行
- 只包含数字和单位缩写的短行（≤10字符）

## 验证方法

1. **运行测试**：
   - 使用包含元数据字段的测试数据
   - 验证不会提取"doi"、"3.4 mi"等无效答案

2. **运行101个样本**：
   - 观察fallback逻辑的提取结果
   - 检查是否还会提取无效答案
   - 验证fallback成功率是否提高

## 注意事项

1. **平衡过滤和提取**：
   - 过滤规则不能太严格，否则可能过滤掉有效答案
   - 需要在过滤无效答案和保留有效答案之间找到平衡

2. **特殊情况处理**：
   - 某些查询可能需要提取DOI或URL（如"这个论文的DOI是什么"）
   - 当前实现优先过滤，如果确实需要这些信息，可以通过其他方式获取

3. **持续优化**：
   - 根据实际运行结果，可能需要调整过滤规则
   - 可以添加更多元数据字段到排除列表
