# 核心系统性能优化紧急方案

**分析时间**: 2025-11-19  
**当前性能**: 平均114.53秒/样本（约2分钟）  
**目标性能**: <30秒/样本  
**性能差距**: 需要提升 **73.8%**

---

## 🔴 当前性能问题

### 性能指标

| 指标 | 当前值 | 目标值 | 差距 |
|------|--------|--------|------|
| 平均推理时间 | 114.53秒 | ≤30秒 | **+84.53秒** |
| 最大处理时间 | 665.74秒 | ≤60秒 | **+605.74秒** |
| 最小处理时间 | 32.36秒 | ≤10秒 | **+22.36秒** |
| 推理效率分数 | 0.38 | ≥0.80 | **-0.42** |

### 性能瓶颈分析

根据代码分析和历史优化报告，主要瓶颈：

1. **LLM API调用时间**（占比：60-80%）
   - 每次LLM调用：5-30秒
   - 每个查询平均：3-5次LLM调用
   - 总耗时：15-150秒

2. **提示词过长**（占比：10-20%）
   - 证据内容过长：14,262字符
   - LLM处理时间长：288.32秒
   - 需要更激进的压缩

3. **串行处理**（占比：10-15%）
   - 证据收集和推理步骤串行
   - LLM调用串行
   - 可以并行化

4. **重复验证**（占比：5-10%）
   - 同一个答案多次验证
   - 可以缓存验证结果

## 🚀 紧急优化方案（P0）

### 优化1: 更激进的证据压缩 ⭐⭐⭐⭐⭐

**问题**: 提示词过长（14,262字符）导致LLM响应慢（288秒）

**优化方案**:
```python
# 在 _calculate_available_evidence_space 中
# 当前：安全边距0.7，最小字符800
# 优化：安全边距0.5，最小字符500

# 在 _process_evidence_for_prompt 中
# 当前：简单查询减少50%，事实查询减少30%
# 优化：简单查询减少70%，事实查询减少50%，复杂查询减少30%
```

**预期效果**: 
- 提示词长度：从14,262字符降至5,000-7,000字符
- LLM响应时间：从288秒降至100-150秒
- **节省时间：100-150秒**

### 优化2: 优化模型选择策略 ⭐⭐⭐⭐⭐

**问题**: 简单查询也使用推理模型，浪费性能

**优化方案**:
```python
# 在 _calculate_task_complexity 中
# 当前：复杂度阈值4才使用推理模型
# 优化：降低阈值到3，更多简单查询使用快速模型

# 在 _select_llm_for_task 中
# 当前：简单查询使用fast模型
# 优化：扩展"简单"定义，包括更多查询类型
```

**预期效果**:
- 快速模型调用：从0%提升到40-50%
- 平均LLM响应时间：从20秒降至8-12秒
- **节省时间：30-50秒**

### 优化3: 减少LLM调用次数 ⭐⭐⭐⭐

**问题**: 每个查询3-5次LLM调用，可以合并或缓存

**优化方案**:
1. **合并相似调用**: 查询类型分析和语义分析合并
2. **缓存验证结果**: 相同答案不重复验证
3. **跳过不必要的验证**: 高置信度答案跳过验证

**预期效果**:
- LLM调用次数：从3-5次降至2-3次
- **节省时间：20-40秒**

### 优化4: 并行处理优化 ⭐⭐⭐

**问题**: 证据收集和推理步骤串行，LLM调用串行

**优化方案**:
1. **证据收集并行**: 多个知识源并行检索
2. **LLM调用并行**: 独立的LLM调用并行执行
3. **批量处理**: 相似查询批量处理

**预期效果**:
- 证据收集时间：从10-20秒降至5-10秒
- **节省时间：5-15秒**

## ⚠️ 风险控制

1. **证据压缩**: 保留最相关的证据，不影响准确率

4. **并行处理**: 不影响结果，只提升速度
