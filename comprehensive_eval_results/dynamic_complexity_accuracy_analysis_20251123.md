## 🔍 问题分析

### 问题1：所有样本都是"medium" ⚠️

**表现**:
- 50个样本中，100%都是"medium"
- 没有"complex"或"simple"样本

**可能原因**:

#### 原因1：评估阈值设置问题

**当前阈值**（证据检索阶段）:
- `complexity_score >= 2.5` → "complex"
- `complexity_score >= 1.0` → "medium"
- `complexity_score < 1.0` → "simple"

**典型样本评分**:
- 证据数6个（> 5）: +1分
- 查询长度（假设>80）: +0.5分
- 查询类型（如果不在complex_types中）: +0分
- 语义复杂度（如果有复杂关键词）: +0.5分

**总评分**: 1.0 - 2.0分 → 全部落在"medium"范围

**分析**:
- 阈值设置导致大部分样本都落在"medium"范围
- 需要调整阈值或评估因素，增加区分度

#### 原因2：使用LLM判断的复杂度

从代码逻辑看，优先使用LLM判断的复杂度：
```python
llm_complexity_for_log = getattr(self, '_last_llm_complexity', None)
if llm_complexity_for_log:
    dynamic_complexity = llm_complexity_for_log
```

**可能情况**:
- LLM判断的复杂度可能都是"medium"
- 或者LLM判断逻辑有问题，导致所有查询都被判断为"medium"

#### 原因3：评估因素不足

**当前评估因素**（证据检索阶段）:
1. 证据数量（权重：2）
2. 证据相关性（权重：1）
3. 查询长度（权重：1）
4. 查询类型（权重：1）
5. 查询语义复杂度（权重：0.5）

**缺失因素**:
- 处理时间（在证据检索阶段未知）
- 推理步骤数（在证据检索阶段未知）
- 重试/fallback情况（在证据检索阶段未知）

**分析**:
- 证据检索阶段的评估因素有限，可能无法准确判断复杂度
- 应该结合后续阶段的评估结果

---

### 问题2：判断结果与实际处理表现不匹配 ⚠️

**表现**:
- 所有样本都是"medium"
- 但实际处理时间差异很大（从6.33秒到565.54秒）
- 处理时间差异约89倍，但复杂度判断相同

**分析**:
- 复杂度判断应该反映实际处理难度
- 处理时间差异如此大，但复杂度判断相同，说明判断不准确

## 🎯 根本原因分析

### 根本原因1：评估时机过早

**问题**:
- 动态复杂度在证据检索阶段就评估了
- 此时还没有处理时间、推理步骤等关键信息
- 只能基于有限的证据信息判断

**影响**:
- 判断不准确
- 无法反映实际处理难度

### 根本原因2：阈值设置不合理

**问题**:
- 阈值设置导致大部分样本都落在"medium"范围
- 缺乏区分度

**当前阈值**:
- `>= 2.5` → "complex"
- `>= 1.0` → "medium"
- `< 1.0` → "simple"

**典型评分**:
- 证据数6个: +1分
- 查询长度>80: +0.5分
- 总评分: 1.0-2.0分 → 全部"medium"

### 根本原因3：评估因素权重不合理

**问题**:
- 证据数量权重过高（+1分）
- 导致所有有6个证据的样本都至少得1分
- 而阈值是1.0，所以全部都是"medium"

---

## 💡 优化建议

### 建议1：调整评估阈值 ⭐⭐⭐⭐⭐

**问题**: 阈值设置导致所有样本都是"medium"

**优化方案**:
```python
# 当前阈值
if complexity_score >= 2.5:
    return "complex"
elif complexity_score >= 1.0:  # 阈值过低
    return "medium"
else:
    return "simple"

# 优化后阈值
if complexity_score >= 3.0:  # 提高complex阈值
    return "complex"
elif complexity_score >= 1.5:  # 提高medium阈值
    return "medium"
else:
    return "simple"
```

**预期效果**:
- 更多样本被判断为"simple"
- 只有真正复杂的样本被判断为"complex"
- 增加区分度

---

### 建议2：调整评估因素权重 ⭐⭐⭐⭐

**问题**: 证据数量权重过高，导致所有样本都至少得1分

**优化方案**:
```python
# 当前逻辑
if evidence_count > 5:  # 默认阈值5.0
    complexity_score += 1  # 权重过高

# 优化后逻辑
if evidence_count > 8:  # 提高阈值
    complexity_score += 2
elif evidence_count > 5:
    complexity_score += 0.5  # 降低权重
```

**预期效果**:
- 只有证据数>8的样本才得高分
- 证据数6个的样本得0.5分，而不是1分
- 增加区分度

---

### 建议3：延迟评估时机 ⭐⭐⭐

**问题**: 在证据检索阶段评估，信息不足

**优化方案**:
- 在最终答案推导阶段评估（此时有处理时间、推理步骤等信息）
- 或者结合多个阶段的评估结果

**预期效果**:
- 评估更准确
- 能反映实际处理难度

---

### 建议4：使用实际处理表现评估 ⭐⭐⭐⭐⭐

**问题**: 当前评估基于证据检索阶段的信息，无法反映实际处理难度

**优化方案**:
- 使用`_assess_complexity_dynamically`方法（基于实际处理表现）
- 在最终答案推导后评估，此时有完整的处理信息

**评估因素**:
- 处理时间（权重：高）
- 推理步骤数（权重：中）
- 重试/fallback情况（权重：中）
- 证据数量（权重：低）

**预期效果**:
- 评估更准确
- 能反映实际处理难度
- 处理时间长的样本会被判断为"complex"
